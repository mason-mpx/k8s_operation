# K8sOperation 平台 — 分支管理与多环境发布流程

> 本文档定义平台代码的分支策略、环境对应关系、合并规范以及完整发布流程。

---

## 一、分支策略总览

### 1.1 分支模型

```
                            ┌───────────────────────────────────────────────────┐
                            │              K8sOperation 分支模型                  │
                            ├───────────────────────────────────────────────────┤
                            │                                                   │
  长期分支                   │   master ──────────────────────────────────▶ 生产  │
                            │     ▲                                             │
                            │     │ MR 合并                                     │
                            │   release ─────────────────────────────▶ 预发布   │
                            │     ▲                                             │
                            │     │ MR 合并                                     │
                            │   test ────────────────────────────▶ 测试环境     │
                            │     ▲                                             │
                            │     │ MR 合并                                     │
                            │   dev ─────────────────────────▶ 开发环境         │
                            │     ▲                                             │
                            │     │ 合并 feature                                │
  短期分支                   │   feature/* ──── 功能开发                          │
                            │   hotfix/*  ──── 紧急修复（从 master 拉出）        │
                            │                                                   │
                            └───────────────────────────────────────────────────┘
```

### 1.2 分支定义

| 分支 | 类型 | 生命周期 | 对应环境 | 保护策略 | 说明 |
|------|------|---------|---------|---------|------|
| `master` | 长期 | 永久 | **生产环境** | 仅允许 MR 合并，禁止直推 | 生产稳定版本，每次合并打 Tag |
| `release` | 长期 | 永久 | **预发布环境** | 仅允许 MR 合并 | 预发布验证，与生产环境配置一致 |
| `test` | 长期 | 永久 | **测试环境** | 仅允许 MR 合并 | QA 测试验证 |
| `dev` | 长期 | 永久 | **开发环境** | 允许直推 / MR | 日常开发集成 |
| `feature/*` | 短期 | 功能完成后删除 | 本地 | 无 | 功能开发分支，从 `dev` 拉出 |
| `hotfix/*` | 短期 | 修复完成后删除 | — | 无 | 紧急修复，从 `master` 拉出 |

## 二、分支与环境对应关系

### 2.1 四环境模型

```
┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│    dev 分支           test 分支          release 分支        master 分支    │
│       │                  │                   │                  │          │
│       ▼                  ▼                   ▼                  ▼          │
│  ┌─────────┐       ┌──────────┐       ┌───────────┐      ┌──────────┐    │
│  │ 🔧 DEV  │──────▶│ 🧪 TEST  │──────▶│ 📦 STAGING│─────▶│ 🚀 PROD  │    │
│  │ 开发环境 │       │ 测试环境  │       │ 预发布环境 │      │ 生产环境  │    │
│  └─────────┘       └──────────┘       └───────────┘      └──────────┘    │
│                                                                            │
│  自动部署             自动部署            手动触发            审批后部署      │
│  宽松管控             中等管控            严格管控            最严管控        │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 各环境管控对比

| 维度 | DEV 开发 | TEST 测试 | STAGING 预发布 | PROD 生产 |
|------|---------|----------|---------------|----------|
| **代码分支** | `dev` | `test` | `release` | `master` |
| **触发方式** | Push 自动构建 | MR 合并自动构建 | 手动触发 | 手动触发 |
| **单元测试** | 可跳过 | 必须执行 | 必须执行 | 必须执行 |
| **代码扫描** | 关闭 | 开启（仅警告） | 开启 + 质量门禁 | 开启 + 严格质量门禁 |
| **人工审批** | 不需要 | 不需要 | 可选 | **强制审批** |
| **部署方式** | 自动部署 | 自动部署 | 自动部署 | 审批通过后自动部署 |
| **回滚支持** | 不需要 | 手动回滚 | 一键回滚 | 一键回滚 + 告警 |
| **通知范围** | 无 | 构建结果 | 全流程通知 | 全流程 + 值班群通知 |
| **发布窗口** | 随时 | 工作时间 | 工作时间 | 工作日 10:00-17:00 |

### 2.3 平台环境标识配置

平台通过流水线的 `deploy_env` 字段区分环境：

```yaml
# 流水线配置示例
Pipeline:
  name: k8s-operation-dev
  deploy_env: dev          # dev / test / staging / prod
  git_branch: dev          # 对应分支
  deploy_cluster: dev-k8s  # 目标集群
  deploy_namespace: k8s-platform-dev
```

## 三、日常开发流程

### 3.1 Feature 功能开发

```
开发者                              Git 仓库                        CI/CD
  │                                    │                              │
  │  1. 从 dev 创建 feature 分支       │                              │
  │───────────────────────────────────▶│                              │
  │  git checkout -b feature/xxx dev   │                              │
  │                                    │                              │
  │  2. 本地开发 & 提交                │                              │
  │───────────────────────────────────▶│                              │
  │                                    │                              │
  │  3. 推送 feature 分支              │                              │
  │───────────────────────────────────▶│                              │
  │                                    │                              │
  │  4. 创建 MR: feature/xxx → dev     │                              │
  │───────────────────────────────────▶│                              │
  │                                    │                              │
  │  5. Code Review（至少 1 人审核）    │                              │
  │◀──────────────────────────────────│                              │
  │                                    │                              │
  │  6. 合并到 dev，删除 feature 分支  │                              │
  │───────────────────────────────────▶│  7. 自动触发 DEV 环境构建    │
  │                                    │─────────────────────────────▶│
  │                                    │                              │
```

### 3.2 Git 操作命令

```bash
# ========== 1. 创建功能分支 ==========
git checkout dev
git pull origin dev
git checkout -b feature/user-auth    # 从 dev 创建功能分支

# ========== 2. 开发 & 提交 ==========
git add .
git commit -m "feat: 添加用户认证模块"
git push origin feature/user-auth

# ========== 3. 创建 MR (在 Gitee/GitHub 上操作) ==========
# 源分支: feature/user-auth → 目标分支: dev
# 指定 Code Review 审核人

# ========== 4. MR 合并后清理 ==========
git checkout dev
git pull origin dev
git branch -d feature/user-auth           # 删除本地分支
git push origin --delete feature/user-auth # 删除远程分支
```

## 四、环境晋升流程

### 4.1 完整晋升链路

```
feature/* ──MR──▶ dev ──MR──▶ test ──MR──▶ release ──MR──▶ master
                   │            │             │               │
                   ▼            ▼             ▼               ▼
              DEV 环境     TEST 环境     STAGING 环境     PROD 环境
              (自动部署)   (自动部署)    (手动触发)       (审批部署)
```

### 4.2 DEV → TEST（开发转测试）

**前提条件**：功能开发完成，DEV 环境自测通过

```bash
# ========== 开发者操作 ==========
# 1. 确保 dev 分支是最新的
git checkout dev
git pull origin dev

# 2. 创建 MR: dev → test (在 Gitee/GitHub 上操作)
#    - 标题: [转测] v1.5.0 用户认证 + 流水线优化
#    - 描述: 列出本次转测包含的功能点
#    - 审核人: 技术负责人

# 3. Code Review 通过后合并
# 4. 合并后自动触发 TEST 环境构建和部署
```

**MR 检查清单**：
- [ ] DEV 环境功能自测通过
- [ ] 编译无错误、无警告
- [ ] 新增接口已补充 Swagger 注解
- [ ] 数据库变更已同步 SQL 脚本

### 4.3 TEST → STAGING（测试转预发布）

**前提条件**：QA 测试通过，无阻塞性 Bug

```bash
# ========== 技术负责人操作 ==========
# 1. 确保 test 分支 QA 验证通过
git checkout test
git pull origin test

# 2. 创建 MR: test → release (在 Gitee/GitHub 上操作)
#    - 标题: [预发布] v1.5.0-rc.1
#    - 描述: 附 QA 测试报告链接
#    - 审核人: 技术负责人 + QA 负责人

# 3. Review 通过后合并
# 4. 在平台手动触发 STAGING 环境流水线
#    - 选择 release 分支
#    - 开启 SonarQube 代码扫描
#    - 开启质量门禁
```

**MR 检查清单**：
- [ ] QA 测试报告签字确认
- [ ] 无 P0/P1 级别 Bug
- [ ] 性能测试通过（如有）
- [ ] 数据库 Migration 脚本准备就绪

### 4.4 STAGING → PROD（预发布转生产）

**前提条件**：预发布环境验收通过，发布窗口内

```bash
# ========== 技术负责人操作 ==========
# 1. 确保 release 分支预发布验证通过
git checkout release
git pull origin release

# 2. 创建 MR: release → master (在 Gitee/GitHub 上操作)
#    - 标题: [发布] v1.5.0
#    - 描述: 版本变更日志 (CHANGELOG)
#    - 审核人: 技术负责人 + 运维负责人（至少 2 人）

# 3. Review 通过后合并

# 4. 打版本 Tag
git checkout master
git pull origin master
git tag -a v1.5.0 -m "Release v1.5.0: 用户认证 + 流水线优化"
git push origin v1.5.0

# 5. 在平台手动触发 PROD 环境流水线
#    - 选择 master 分支
#    - 强制开启 SonarQube + 质量门禁
#    - 触发后需审批人批准才执行部署
```

**MR 检查清单**：
- [ ] STAGING 环境验收签字
- [ ] SonarQube 质量门禁 OK
- [ ] CHANGELOG 已更新
- [ ] 回滚方案已准备
- [ ] 发布时间在工作日 10:00-17:00 内
- [ ] 值班人员已通知

## 五、Hotfix 紧急修复流程

### 5.1 流程说明

生产环境出现紧急 Bug 时，从 `master` 拉出 `hotfix/*` 分支，修复后同时合并回 `master` 和 `dev`。

```
master ─────────────┬──────────────────────────────────┬─────────▶
                    │                                  │
                    │  git checkout -b hotfix/fix-xxx   │  MR 合并
                    ▼                                  ▲
              hotfix/fix-xxx ──── 修复 ──── 测试 ──────┘
                    │
                    │  同步合并回 dev（防止修复丢失）
                    ▼
dev ───────────────────────────────────────────────────────────▶
```

### 5.2 Hotfix 操作步骤

```bash
# ========== 1. 创建 hotfix 分支（从 master 拉出） ==========
git checkout master
git pull origin master
git checkout -b hotfix/fix-login-crash

# ========== 2. 修复代码 & 提交 ==========
git add .
git commit -m "hotfix: 修复登录页面崩溃问题"
git push origin hotfix/fix-login-crash

# ========== 3. 创建 MR: hotfix → master ==========
# 审核人: 技术负责人 + 运维负责人
# 标签: hotfix / 紧急

# ========== 4. Review 通过后合并到 master ==========
# 平台触发 PROD 环境构建（走审批流程）

# ========== 5. 打补丁版本 Tag ==========
git checkout master
git pull origin master
git tag -a v1.5.1 -m "Hotfix v1.5.1: 修复登录页面崩溃"
git push origin v1.5.1

# ========== 6. 同步合并回 dev（重要！防止修复丢失） ==========
git checkout dev
git pull origin dev
git merge master
git push origin dev

# ========== 7. 清理 ==========
git branch -d hotfix/fix-login-crash
git push origin --delete hotfix/fix-login-crash
```

### 5.3 Hotfix 特殊规则

| 规则 | 说明 |
|------|------|
| 从 `master` 拉出 | 确保基于生产代码修复 |
| 合并回 `master` + `dev` | 防止修复在后续版本丢失 |
| 可跳过 STAGING | 紧急情况下可直接发生产，但需额外审批 |
| 必须打补丁 Tag | 如 v1.5.1（主版本.次版本.补丁号） |
| 发布窗口可例外 | 紧急修复不受发布窗口限制 |

| `ENABLE_ARTIFACT_UPLOAD` | `false` | `false` | `true` | `true` |
| `require_approval` | `false` | `false` | `false` | `true` |
| 目标集群 | dev-cluster | test-cluster | staging-cluster | prod-cluster |
| 目标命名空间 | `k8s-platform-dev` | `k8s-platform-test` | `k8s-platform-staging` | `k8s-platform` |
| 副本数 | 1 | 1 | 2 | 2+ |

## 七、完整发布流程图

### 7.1 标准版本发布（v1.5.0）

```
时间轴
──────────────────────────────────────────────────────────────────────▶

Sprint 开发（1-2 周）
│
├─ Day 1-10: 功能开发
```

## 八、环境配置差异

### 8.1 config.yaml 各环境关键差异

```yaml
# ==================== DEV 环境 ====================
Server:
  RunMode: debug              # debug 模式，详细日志
  Port: 8080
App:
  LogLevel: debug
  ErrorCode:
    AllowOverride: true       # 允许错误码覆盖（开发调试）

# ==================== TEST 环境 ====================
Server:
  RunMode: release
  Port: 8080
App:
  LogLevel: info
  ErrorCode:
    AllowOverride: true       # 测试也允许

# ==================== STAGING 环境 ====================
Server:
  RunMode: release
  Port: 8080
App:
  LogLevel: info
  ErrorCode:
    AllowOverride: false      # 与生产一致

# ==================== PROD 环境 ====================
Server:
  RunMode: release
  Port: 8080
App:
  LogLevel: warn              # 只记录警告及以上
  ErrorCode:
    AllowOverride: false      # 禁止覆盖
  LogMaxSize: 100             # 100MB 日志轮转
  LogMaxBackup: 10            # 保留 10 个备份
  LogMaxAge: 90               # 保留 90 天
```

## 九、版本号规范

### 9.1 语义化版本（SemVer）

```
v{MAJOR}.{MINOR}.{PATCH}

v1.5.0
│ │ └── 补丁号：Hotfix 修复 +1
│ └──── 次版本：新功能、向后兼容 +1
└────── 主版本：不兼容的重大变更 +1
```

### 9.2 Tag 规范

| 场景 | Tag 格式 | 示例 |
|------|---------|------|
| 正式发布 | `v{MAJOR}.{MINOR}.{PATCH}` | `v1.5.0` |
| Hotfix | `v{MAJOR}.{MINOR}.{PATCH}` | `v1.5.1` |
| 预发布 | `v{MAJOR}.{MINOR}.{PATCH}-rc.{N}` | `v1.5.0-rc.1` |

### 9.3 Commit 规范

```
<type>: <description>

type 取值:
  feat:     新功能
  fix:      Bug 修复
  hotfix:   紧急修复
  docs:     文档更新
  refactor: 重构
  perf:     性能优化
  test:     测试用例
  chore:    构建/工具变更
```

## 十、操作速查表

### 10.1 日常开发

```bash
# 创建功能分支
git checkout dev && git pull origin dev
git checkout -b feature/xxx

# 开发完成，合并到 dev
git push origin feature/xxx
# → Gitee 创建 MR: feature/xxx → dev → Review → 合并
```

### 10.2 环境晋升

```bash
# DEV → TEST
# → Gitee 创建 MR: dev → test → Review → 合并 → 自动构建

# TEST → STAGING
# → Gitee 创建 MR: test → release → Review → 合并
# → 平台手动触发 STAGING 流水线

# STAGING → PROD
# → Gitee 创建 MR: release → master → Review → 合并
# → git tag v1.x.x && git push origin v1.x.x
# → 平台手动触发 PROD 流水线 → 等待审批
```

### 10.3 紧急修复

```bash
# 从 master 拉 hotfix
git checkout master && git pull origin master
git checkout -b hotfix/fix-xxx

# 修复后合并到 master
# → MR: hotfix/fix-xxx → master → 审批 → 触发 PROD 流水线

# 同步到 dev
git checkout dev && git merge master && git push origin dev
```

## 十一、注意事项

### 11.1 合并方向铁律

```
feature/* → dev → test → release → master    ✅ 正确方向
master → dev                                   ✅ Hotfix 同步
master → release → test → dev                  ❌ 禁止反向合并
```

### 11.2 关键原则

| 原则 | 说明 |
|------|------|
| **镜像不变性** | 同一镜像（image@sha256:digest）跨环境晋升，不重复构建 |
| **代码不变性** | 合并到上游的代码不允许再修改，只能通过新 MR 修复 |
| **环境隔离** | 各环境独立集群/命名空间/数据库，互不影响 |
| **禁止跳级** | 不允许从 dev 直接合并到 release 或 master |
| **Hotfix 例外** | 紧急修复可从 master 拉出，修复后直接合并回 master |
| **Tag 必打** | 每次生产发布必须打 Tag，用于回滚和追溯 |

### 12.2 阶段定义

| 序号 | 阶段类型 | 阶段名称 | 默认启用 | 执行方 | 说明 |
|:----:|----------|----------|:--------:|:------:|------|
| 1 | `clean` | 清理工作空间 | ✅ | Jenkins | 清理 workspace + 拉取代码 |
| 2 | `checkout` | 代码检出 | ✅ | Jenkins | Git 代码拉取，采集 commit 信息 |
| 3 | `dependencies` | 依赖下载 | ✅ | Jenkins | go mod / mvn / npm / pip 依赖下载 |
| 4 | `compile` | 编译检查 | ✅ | Jenkins | 编译源代码，产出二进制/JAR/dist |
| 5 | `test` | 单元测试 | ✅ | Jenkins | go test / mvn test / pytest / npm test |
| 6 | `lint` | 代码检查 | ✅ | Jenkins | golangci-lint / flake8 / eslint |
| 7 | `sonar` | SonarQube 代码扫描 | ❌ | Jenkins | SonarQube 代码质量扫描（`EnableSonar` 控制） |
| 8 | `quality_gate` | 质量门禁检查 | ❌ | Jenkins | 等待 SonarQube 质量门禁结果 |
| 9 | `build_binary` | 构建制品 | ✅ | Jenkins | 编译产出最终制品（二进制/JAR/dist） |
| 10 | `upload_artifact` | 上传制品库 | ❌ | Jenkins | 将制品上传到平台制品库（`ENABLE_ARTIFACT_UPLOAD` 控制） |
| 11 | `build` | 打包镜像 | ✅ | Jenkins | nerdctl build + BuildKit 层缓存 |
| 12 | `push` | 推送镜像 | ✅ | Jenkins | nerdctl push --concurrency 8 到 Harbor |
| 13 | `approval` | 人工审批 | ❌ | 平台 | 生产环境强制审批（`RequireApproval` 控制） |
| 14 | `deploy` | 部署 | ❌ | 平台 | K8s Deployment/StatefulSet/DaemonSet 滚动更新 |

| Reliability Rating | 可靠性评级 (A-E) | ❌ |
